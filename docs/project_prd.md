# 生鲜仓储系统 Demo（Fresh Warehouse Demo）

## 1. 项目概述（Project Overview）

本项目是一个用于演示的仓储管理系统（Warehouse Management System, WMS）Demo，主要用于模拟生鲜供应链企业的基础仓储管理流程。

系统的核心目标是记录和管理商品在仓库中的库存变化，包括入库、出库以及库存查询等基本操作。

通过该系统，可以清晰地了解：

- 当前有哪些商品
- 商品存放在哪个仓库
- 每个商品当前库存是多少
- 每一次库存变化的记录

本 Demo 将模拟一个简化的供应链业务流程：

采购入库 → 更新库存 → 客户订单出库 → 库存减少 → 记录库存流水

本系统不会在第一阶段实现复杂功能（如成本核算、价格策略、智能补货等），而是专注于构建一个完整、清晰、可运行的库存管理闭环，用于演示仓储系统的核心逻辑。
## 2. 系统范围（System Scope）

本系统为一个简化版仓储管理系统 Demo，第一阶段仅实现最核心的仓储业务流程，不包含复杂的企业级功能。

系统主要包含以下模块：

### 2.1 商品管理（Product Management）

用于管理系统中的商品信息。

主要功能：

- 新增商品
- 查看商品列表
- 删除商品

每个商品包含基本信息，例如：

- 商品名称
- 商品 SKU（唯一编号）

---

### 2.2 仓库管理（Warehouse Management）

用于管理仓库信息。

主要功能：

- 新增仓库
- 查看仓库列表

每个仓库包含：

- 仓库名称
- 仓库编号

---

### 2.3 库存查询（Inventory Management）

用于查看当前库存情况。

系统需要能够展示：

- 商品名称
- 所在仓库
- 当前库存数量

用户可以通过库存页面快速了解每个商品在不同仓库中的库存情况。

---

### 2.4 入库管理（Inbound Management）

用于记录商品进入仓库的操作。

入库操作需要记录：

- 商品
- 仓库
- 入库数量
- 操作时间

入库完成后，系统需要自动增加对应库存。

---

### 2.5 出库管理（Outbound Management）

用于记录商品从仓库发出的操作。

出库操作需要记录：

- 商品
- 仓库
- 出库数量
- 操作时间

出库完成后，系统需要自动减少库存。

系统需要保证：

- 库存不能为负数
- 如果库存不足，系统需要提示错误。

---

### 2.6 库存流水（Stock Movement Logs）

系统需要记录每一次库存变化。

每一条库存流水需要包含：

- 商品
- 仓库
- 操作类型（入库 / 出库）
- 变动数量
- 操作时间

通过库存流水，用户可以追溯每一次库存变化的来源。
## 3. 系统流程（System Workflow）

本系统围绕库存变化这一核心逻辑运行。  
所有库存变化都必须通过入库或出库操作完成，并记录库存流水。

系统的基本流程如下：

商品创建 → 仓库创建 → 商品入库 → 更新库存 → 商品出库 → 更新库存 → 记录库存流水

---

### 3.1 商品与仓库初始化

在进行库存操作之前，系统需要先完成基础数据的创建：

1. 创建商品（Product）
2. 创建仓库（Warehouse）

只有存在的商品和仓库才能进行库存操作。

---

### 3.2 入库流程（Inbound Workflow）

入库表示商品进入仓库。

入库流程：

1. 用户选择商品
2. 用户选择仓库
3. 输入入库数量
4. 系统增加库存
5. 系统记录一条库存流水（类型：入库）

入库完成后，库存数量增加。

---

### 3.3 出库流程（Outbound Workflow）

出库表示商品从仓库发出，例如发货给客户。

出库流程：

1. 用户选择商品
2. 用户选择仓库
3. 输入出库数量
4. 系统检查库存是否充足
5. 如果库存充足，则减少库存
6. 系统记录一条库存流水（类型：出库）

如果库存不足，系统需要提示错误，并拒绝出库操作。

---

### 3.4 库存更新逻辑

库存的变化规则如下：

入库操作：

库存 = 当前库存 + 入库数量

出库操作：

库存 = 当前库存 - 出库数量

系统需要确保库存始终大于或等于 0。

---

### 3.5 库存流水记录

系统需要记录每一次库存变化。

库存流水包含以下信息：

- 商品
- 仓库
- 操作类型（入库 / 出库）
- 变动数量
- 操作时间

库存流水用于追踪库存历史变化，方便审计和问题排查。
## 4. 数据模型（Data Model）

为了支持仓储系统的基本功能，系统需要设计以下核心数据表。

---

### 4.1 商品表（products）

用于存储商品的基本信息。

字段：

- id：商品ID（主键）
- name：商品名称
- sku：商品唯一编号
- created_at：创建时间

示例：

| id | name | sku | created_at |
|----|------|-----|------------|
| 1 | 苹果 | APPLE001 | 2026-03-10 |

---

### 4.2 仓库表（warehouses）

用于存储仓库信息。

字段：

- id：仓库ID（主键）
- name：仓库名称
- code：仓库编号
- created_at：创建时间

示例：

| id | name | code |
|----|------|------|
| 1 | 成都仓 | CD01 |

---

### 4.3 库存表（inventory）

用于记录商品在每个仓库中的库存数量。

字段：

- id：记录ID
- product_id：商品ID
- warehouse_id：仓库ID
- quantity：当前库存数量
- updated_at：更新时间

示例：

| product_id | warehouse_id | quantity |
|-------------|--------------|---------|
| 1 | 1 | 70 |

表示：

商品：苹果  
仓库：成都仓  
库存：70

---

### 4.4 库存流水表（stock_movements）

用于记录每一次库存变化。

字段：

- id：流水ID
- product_id：商品ID
- warehouse_id：仓库ID
- type：操作类型（inbound / outbound）
- quantity：变动数量
- created_at：操作时间

示例：

| product_id | warehouse_id | type | quantity |
|-------------|--------------|------|---------|
| 1 | 1 | inbound | 100 |
| 1 | 1 | outbound | 30 |

通过库存流水，可以追踪库存变化历史。
## 5. API 设计（API Design）

为了支持前端页面与后端系统通信，系统需要提供一组 REST API 接口。

以下是第一阶段需要实现的核心接口。

---

### 5.1 商品接口（Products API）

获取商品列表

GET /products

创建商品

POST /products

请求示例：

{
  "name": "苹果",
  "sku": "APPLE001"
}

---

### 5.2 仓库接口（Warehouses API）

获取仓库列表

GET /warehouses

创建仓库

POST /warehouses

请求示例：

{
  "name": "成都仓",
  "code": "CD01"
}

---

### 5.3 库存接口（Inventory API）

获取库存列表

GET /inventory

返回示例：

[
  {
    "product": "苹果",
    "warehouse": "成都仓",
    "quantity": 70
  }
]

---

### 5.4 入库接口（Inbound API）

用于增加库存。

POST /inbound

请求示例：

{
  "product_id": 1,
  "warehouse_id": 1,
  "quantity": 100
}

---

### 5.5 出库接口（Outbound API）

用于减少库存。

POST /outbound

请求示例：

{
  "product_id": 1,
  "warehouse_id": 1,
  "quantity": 30
}

系统需要检查库存是否充足，如果库存不足需要返回错误。

---

### 5.6 库存流水接口（Stock Movements API）

获取库存变动记录。

GET /stock-movements
